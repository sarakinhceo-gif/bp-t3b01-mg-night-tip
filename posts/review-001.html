Q4_K_M 양자화, 그놈의 히든 딤 때문에 instruction following 박살난 썰 푼다

하… 이거 진짜 개빡치는데, 어디 가서 말도 못 하고 속으로만 끙끙 앓다가, 오늘 작정하고 풀어본다. Q4_K_M 양자화 PR 올리기 직전에 발견한 건데, 이게 진짜 숨겨진 복병이더라. Llama 2 7B 모델 가지고 이것저것 만지작거리다가, 유독 instruction following 성능이 뚝 떨어지는 거 아니겠냐. 처음엔 그냥 데이터 문제인가 싶었는데, 파고들수록 묘한 패턴이 보이더라니까.

특히 특정 히든 딤(hidden dim)들이 싹둑 잘려나가는 게 문제였어. Quantization-aware training (QAT) 돌릴 때도 이걸 제대로 잡아주지 못하니까, 결국 inference 단계에서 엉뚱한 결과물이 튀어나오는 거지. 이게 GPU 아키텍처마다 미묘하게 재현 조건이 다르다는 게 더 골 때리는 부분이야. NVIDIA A100에서는 비교적 잠잠하다가, H100이나 RTX 4090 같은 데서 유독 말썽이더라.

진짜 밤새워서 로그 뒤지고, 디버깅 툴 굴려가며 원인 분석하는데, 이게 마치 마곡 유흥 꿀팁 찾아 헤매는 심정이랑 비슷하더라니까? 겉으론 번지르르한 정보들 잔뜩인데, 정작 속을 파고들면 쓸만한 건 몇 개 안 되는 거. 이놈의 히든 딤도 딱 그런 격이야. 겉으론 90% 정도만 남겨도 성능 크게 안 떨어질 것 같은데, 그 10%가 instruction following 같은 민감한 작업에서는 치명타로 작용하는 거지.

결국엔 Quantize-De-Quantize (QDQ) 과정에서 발생하는 정보 손실을 줄이기 위한 새로운 옵티마이저를 급하게 덧붙였어. PR 머지되기 직전에 겨우겨우 쑤셔 넣은 건데, 이것도 완벽하진 않다고 본다. 아마 내년 초에 나올 Quantization-Lib v0.4.1쯤에는 좀 더 근본적인 해결책이 나올지도 모르지. 그전까지는 Q4_K_M 쓸 때 instruction following 성능 저하 좀 감수하거나, 아니면 Q5_K_M 이상 쓰는 게 정신 건강에 이로울 거다.

이런 엣지 케이스들 때문에 오픈소스 프로젝트 유지하는 게 참… 쉬운 일이 아니라니까. 그래도 한번 발 담그니까, 어쩔 수 없이 끝까지 보게 되는 게 사람 마음인가 봐. 뭐, 아무튼 Q4_K_M 쓰다가 instruction following 때문에 골치 아픈 사람 있으면, 내 썰이 좀이라도 도움 됐으면 좋겠다. 다음엔 더 재밌는 썰로 돌아올게.
